# RemUp 更新日志 v4.1

## 🎉 新增功能

### 1. 四级标题支持（次级卡片）✨

现在可以使用 `####` 在卡片内部创建更细的分组：

```remup
## 主卡片
[@重要]

### 区域一

#### 次级卡片 1 - 基础概念
[@基础]

这是次级卡片的内容 ^用于细分知识点

#### 次级卡片 2 - 进阶应用
[@进阶]

另一个分组的内容 ^灵活组织

/+>
```

**使用场景**：
- 复杂知识点的分层组织
- 同一主题下的不同方面
- 对比相关但不同的概念
- 逐步深入的知识讲解

### 2. 默认卡片自动生成 🎴

现在可以在归档下直接写内容，无需显式创建卡片：

```remup
# 归档名

[@重要]

---区域名

内容可以直接写在区域下 ^自动创建默认卡片

/+>
```

这会让极简语法更加简洁！编译时会自动将归档下的第一个区域前的内容包装到"默认卡片"中。

## 🐛 Bug 修复

### 1. 卡片结束识别问题 ✅

**问题描述**：之前版本中，当使用极简语法（`##` 创建卡片）时，有时会出现卡片无法正确结束的问题，导致后续卡片无法被识别。

**根本原因**：
1. 词法分析器正确识别了 `CARD_END` 标记
2. 但语法分析器在处理区域内容时，遇到 `CARD_END` 会退出循环但不消费该标记
3. 虽然在 parse_card 最后会消费 `CARD_END`，但当有多张卡片时，逻辑不够清晰

**解决方案**：
- 优化了 parse() 方法的逻辑，增加了对默认卡片的支持
- 改进了卡片和区域的归属关系处理
- 确保所有 CARD_END 标记都被正确消费

**测试验证**：
- 多卡片场景：✅ 正常工作
- 混合语法场景：✅ 正常工作  
- 默认卡片场景：✅ 正常工作
- 次级卡片场景：✅ 正常工作

### 2. 归档下直接写内容的支持 ✅

**问题描述**：用户期望在 `# 归档`后直接写 `### 区域` 能够工作，但之前版本要求必须有`## 卡片`。

**解决方案**：
- 当在归档下直接遇到 REGION 时，自动创建"默认卡片"
- 所有该归档下的区域都会添加到这个默认卡片中
- 直到遇到下一个显式的 `## 卡片` 或下一个`# 归档`

## 📝 语法总结

现在 RemUp 支持以下层级结构：

```
# 归档名          ← 第一级：大主题分组
  [默认卡片]       ← 隐含的第二级（可选）
    ### 区域       ← 第三级：知识的某个方面
  
  ## 卡片主题      ← 第二级：知识点单元（显式）
    [@标签]
    ### 区域       ← 第三级
      #### 次级卡片 ← 第四级：更细的分组（新增）
```

## 🎯 使用建议

### 何时使用默认卡片？

当你想要快速记录，不需要为卡片单独命名时：

```remup
# 临时笔记

[@想法]

---灵感 1
突然想到的点子...

---灵感 2
另一个想法...

/+>
```

### 何时使用显式卡片？

当你需要明确的主题和组织结构时：

```remup
# Python 学习

## 变量和作用域
[@核心] [@重要]

### 基本概念
...

### 实战应用
...

## 函数和类
[@进阶]
...

/+>
```

### 何时使用次级卡片？

当卡片内的内容需要进一步分组时：

```remup
## 设计模式
[@高级]

### 创建型模式

#### 单例模式
保证一个类只有一个实例...

#### 工厂模式
定义创建对象的接口...

### 结构型模式

#### 适配器模式
将一个类的接口转换成...

#### 装饰器模式
动态地给对象添加职责...

/+>
```

---

## ⚠️ 注意事项

1. **不要混用层级**：避免在显式卡片和默认卡片之间来回切换
2. **次级卡片不宜过多**：每个卡片内建议不超过 3-5 个次级卡片
3. **保持结构清晰**：虽然灵活，但仍需注意逻辑层次
